## 2. Project Overview

Система решает проблему упущенных выгодных объявлений о продаже б/у автомобилей: хорошие предложения уходят за минуты, а ручной мониторинг Avito и auto.ru в режиме 24/7 невозможен. CarWatch Bot — это персональный Telegram-бот для одного владельца, который периодически и человекоподобно опрашивает auto.ru и Avito по заданным фильтрам и присылает новые подходящие объявления без дублей. Реализуй как Modular Monolith на Python (один процесс). В scope MVP входят: бот с командами управления, до 3 конфигураций фильтров, обе площадки, дедупликация уведомлений, запуск и на сервере (RU IP), и локально.

> Аудитория этого раздела: AI-агент, который должен понять контекст задачи перед генерацией кода.

## 3. Target Audience & User Scenarios

#### Владелец (единственный пользователь)
- **Роль:** частный покупатель авто, владелец сервера, управляет ботом командами.
- **Основная задача:** получать свежие подходящие объявления раньше других покупателей.
- **Ключевые сценарии использования:**
  1. Создаёт конфигурацию фильтра командой `/add` (марка+модель, год, пробег обязательны; цена, цвет и пр. — опционально).
  2. Получает в Telegram карточку нового объявления с обеих площадок.
  3. Просматривает/удаляет/ставит на паузу свои конфигурации (`/list`, `/del`, `/pause`, `/resume`, `/status`).

**Жёсткое ограничение:** бот обслуживает ТОЛЬКО владельца. Все сообщения от chat_id, не равного `OWNER_CHAT_ID`, игнорируются.

## 4. MVP Scope

### 4.1 Что входит в MVP

- [ ] Telegram-бот (single-owner, whitelist по `OWNER_CHAT_ID`)
- [ ] Источник auto.ru (HTTP+JSON сбор)
- [ ] Источник Avito (браузерный сбор, best-effort)
- [ ] До 3 конфигураций фильтров, управление командами `/add /list /del /pause /resume /status`
- [ ] Обязательные фильтры: марка+модель, год (от/до), пробег (от/до)
- [ ] Опциональные фильтры: цена, число владельцев, цвет, коробка, топливо, регион/город, привод, объём двигателя
- [ ] Периодический опрос с джиттером и человекоподобными паузами
- [ ] Дедупликация (одно объявление — одно уведомление за всё время)
- [ ] Уведомление-карточка: заголовок, цена, год, пробег, ключевые параметры, прямая ссылка
- [ ] Двойной режим запуска: сервер (RU IP) и локально, через профиль окружения

### 4.2 Что исключено из MVP

- Веб-интерфейс — исключён, потому что управление через команды бота достаточно.
- Мультипользовательский режим / биллинг — исключён, потому что инструмент личный.
- Drom.ru и прочие площадки — исключены, потому что не раздуваем scope (vNext).
- ML-оценка выгодности цены — исключена, потому что это отдельная продуктовая гипотеза.
- История цен и аналитика — исключены из первой версии.

> Без явного списка исключений агент может реализовать лишнее. Не реализуй перечисленное выше.

## 5. System Architecture

### 5.1 Архитектурный стиль

**Modular Monolith** — один Python-процесс с модулями (scrapers, scheduler, bot, storage, core). Обоснование: personal tool на 1 пользователя, нет нагрузки и потребности в распределённости; простота напрямую повышает стабильность, что является приоритетом заказчика.

### 5.2 High-Level Architecture

```
Telegram client (owner)
        │  команды / уведомления
┌───────▼────────────────────────────────────┐
│ BOT MODULE (aiogram, long-polling, whitelist)│
│ /add /list /del /pause /resume /status       │
└───────┬─────────────────────────────────────┘
        │
   ┌────┼─────────────────────┬──────────────────┐
┌──▼─────────┐      ┌──────────▼───────┐   ┌──────▼────────┐
│ SCHEDULER  │◄────►│ STORAGE (SQLite) │◄─►│ NOTIFIER       │
│ APScheduler│      │ configs/seen/    │   │ карточки       │
│ jitter+паузы│     │ state            │   │                │
└──┬─────────┘      └──────────────────┘   └───────────────┘
   │ опрос по каждому активному конфигу
┌──▼──────────────────────────────────────────────────────┐
│ SCRAPER LAYER: Scraper.search(config) -> list[Listing]   │
│ ┌────────────────────┐     ┌──────────────────────────┐  │
│ │ AutoRuScraper      │     │ AvitoScraper             │  │
│ │ httpx + curl_cffi  │     │ Playwright (Chromium)    │  │
│ │ human-like delays  │     │ human-like, best-effort  │  │
│ └────────────────────┘     └──────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 5.3 Компоненты системы

#### Bot Module
- **Тип:** Frontend (Telegram) / контроллер
- **Ответственность:** приём команд, валидация владельца, диалог создания конфига, отправка уведомлений.
- **Технологический стек:** Python, aiogram (async), long-polling.
- **Взаимодействует с:** Storage, Notifier.
- **Критические ограничения:** только long-polling (не webhook), чтобы работать и локально, и на сервере без белого IP/домена. Whitelist по `OWNER_CHAT_ID`.

#### Scheduler
- **Тип:** Backend Service
- **Ответственность:** запуск опроса по каждому активному конфигу с рандомизированным интервалом (джиттер) и паузами между конфигами.
- **Технологический стек:** APScheduler (async).
- **Взаимодействует с:** Scraper Layer, Storage, Notifier.
- **Критические ограничения:** конфиги опрашиваются последовательно, не одновременно; параллелизм браузерного сбора = 1.

#### Scraper Layer
- **Тип:** Backend Service
- **Ответственность:** сбор объявлений по фильтрам с площадок через единый интерфейс `Scraper.search(config) -> list[Listing]`.
- **Технологический стек:** httpx + curl_cffi (auto.ru), Playwright Chromium (Avito).
- **Взаимодействует с:** внешние площадки, core (фильтрация).
- **Критические ограничения:** человекоподобное поведение обязательно; лимит 1–2 страниц на опрос.

#### Storage
- **Тип:** Database
- **Ответственность:** хранение конфигов, виденных объявлений, служебного состояния.
- **Технологический стек:** SQLite (один файл).
- **Взаимодействует с:** все модули.
- **Критические ограничения:** один процесс, нет конкурентного доступа.

#### Notifier
- **Тип:** Backend Service
- **Ответственность:** форматирование карточки объявления и отправка через Bot Module.
- **Технологический стек:** Python (aiogram send).

## 6. Frontend Architecture

Отдельного frontend-слоя нет. UX = команды Telegram-бота (соответствует MVP: веб-интерфейс исключён).

#### 6.3 Команды (аналог "страниц")

- `/add` — диалог создания конфигурации. Сначала обязательные поля (марка+модель, год от/до, пробег от/до), затем опциональные. Отказ при отсутствии обязательных или при достижении лимита в 3 конфигурации.
- `/list` — список конфигураций с их параметрами и статусом (active/paused).
- `/del N` — удалить конфигурацию N.
- `/pause N` / `/resume N` — приостановить/возобновить опрос конфигурации.
- `/status` — состояние каналов (auto.ru / Avito), время последнего успешного опроса по каждому, число активных конфигов.

Состояния диалога `/add`: пошаговый сбор полей с валидацией на каждом шаге.

## 7. Backend Architecture

### 7.1 Технологический стек

```
- Runtime / Language: Python 3.11+
- Async-ядро: asyncio
- Telegram: aiogram (long-polling)
- Планировщик: APScheduler (async)
- HTTP-сбор: httpx + curl_cffi (TLS-имперсонизация браузера)
- Браузерный сбор: Playwright (Chromium)
- Тестирование: pytest
```

Все компоненты — open-source и бесплатны (требование заказчика).

### 7.2 Структура директорий

```
carbot/
  bot/          # хендлеры команд, форматирование карточек
  scheduler/    # планировщик, джиттер, паузы
  scrapers/     # base.py, autoru.py, avito.py
  storage/      # модели, миграции, доступ к SQLite
  core/         # config-loader, фильтрация, дедупликация, модель Listing
  config/       # .env.local | .env.server
  main.py
```

### 7.3 Сервисная модель

Единый интерфейс сбора:

```python
class Scraper:
    source: str  # "autoru" | "avito"
    async def search(self, config: Config) -> list[Listing]:
        ...
```

`AutoRuScraper` и `AvitoScraper` реализуют этот интерфейс. Scheduler не знает деталей конкретной площадки.

### 7.4 Business Logic

- **Сборка запроса:** обязательные/жёсткие фильтры (марка+модель, год, пробег + любые заданные опциональные, которые поддерживает площадка) → транслируются в параметры запроса/навигации.
- **Пост-фильтрация:** опциональные фильтры, которые нельзя задать на площадке, применяются к результатам в `core`.
- **Дедупликация:** ключ `(source, listing_id)`; перед уведомлением проверка `is_seen`, после отправки — `mark_seen`.
- **Human-like:** рандомизированные задержки между запросами/действиями, джиттер интервала опроса, паузы между конфигами, ограничение страниц.

### 7.5 [MISSING: точные внутренние эндпоинты/параметры запросов auto.ru и DOM-селекторы Avito]
Конкретные URL внутренних эндпоинтов auto.ru, схема их JSON-ответа и CSS/DOM-селекторы карточек Avito не зафиксированы на этапе проектирования (внешние, изменчивые). Их необходимо определить эмпирически на этапе T4.3 / T6.2–T6.3 через DevTools браузера и сохранить как фикстуры для тестов. ### 10.4 Environment Variables

| Переменная | Описание | Обязательная |
|-----------|---------|--------------|
| TELEGRAM_BOT_TOKEN | токен Telegram-бота | Yes |
| OWNER_CHAT_ID | chat_id владельца (whitelist) | Yes |
| PROFILE | local / server | Yes |
| DB_PATH | путь к carbot.db | Yes |
| AVITO_ENABLED | true/false (feature flag) | No |
| PROXY_URL | прокси. **Обязателен при PROFILE=local, если локальный IP не российский.** Для PROFILE=server (RU IP) не требуется | Conditional |
| POLL_INTERVAL_MIN | базовый интервал опроса, мин | No (default 30) |
| POLL_JITTER_MIN | разброс джиттера, мин (±) | No (default 7) |
| CONFIG_GAP_SEC_MIN | мин. пауза между конфигами, сек | No (default 30) |
| CONFIG_GAP_SEC_MAX | макс. пауза между конфигами, сек | No (default 90) |
| ACTION_DELAY_SEC_MIN | мин. задержка между браузерными действиями, сек | No (default 1.5) |
| ACTION_DELAY_SEC_MAX | макс. задержка между браузерными действиями, сек | No (default 4) |
| HTTP_DELAY_SEC_MIN | мин. задержка между HTTP-запросами, сек | No (default 2) |
| HTTP_DELAY_SEC_MAX | макс. задержка между HTTP-запросами, сек | No (default 5) |
| MAX_PAGES_PER_POLL | сколько страниц листинга обходить за опрос | No (default 2) |

#### Тайминги: пояснение для разработчика

В системе три независимых тайминга, не один:

1. **Poll interval** (`POLL_INTERVAL_MIN` ± `POLL_JITTER_MIN`) — как часто бот идёт проверять новые объявления по конфигу. Определяет свежесть уведомлений. Default 30 ± 7 → фактический интервал случайно в диапазоне **23–37 мин**. Джиттер обязателен: ровный «каждые 30:00» детектируется как бот.
2. **Config gap** (`CONFIG_GAP_SEC_MIN..MAX`) — случайная пауза между опросами разных конфигов (30–90 сек), чтобы 2–3 конфига не ударили по площадке одновременно. На свежесть не влияет.
3. **Action/HTTP delay** (`ACTION_DELAY_*`, `HTTP_DELAY_*`) — случайные паузы между переходами/скроллом/кликами (браузер, Avito) и между HTTP-запросами (auto.ru). Чисто human-like маскировка внутри одной сессии сбора. `MAX_PAGES_PER_POLL=2`: новые объявления всегда на первой странице сортировки «по дате», глубже лезть не нужно и подозрительно.

Все диапазоны — случайная равномерная выборка при каждом срабатывании (не фиксированное значение).

## 11. Observability

### 11.1 Logging
Логировать: старт/стоп опросов, число найденных/новых объявлений, ошибки скраперов, события блокировки/капчи, отправленные уведомления. Уровень логов — по профилю (local: DEBUG, server: INFO). Без секретов.

Время последнего успешного опроса по каждому каналу; число новых объявлений за опрос; счётчик ошибок/блокировок.

### 11.3 Health Checks
Команда `/status`: для каждого канала — last success time + состояние (ok / degraded). Avito = degraded при капче/блоке, без падения процесса.

## 15. Architecture Decision Records (ADR)

#### ADR-1: Modular Monolith вместо микросервисов
- **Статус:** Accepted
- **Контекст:** personal tool на 1 пользователя, без нагрузки.
- **Решение:** один Python-процесс с модулями.
- **Альтернативы:** микросервисы + брокер.
- **Последствия:** нельзя независимо масштабировать модули (не требуется).

#### ADR-2: Гибридный сбор — HTTP для auto.ru, браузер для Avito
- **Статус:** Accepted
- **Контекст:** нет бесплатного API; разный уровень защиты площадок.
- **Решение:** httpx+curl_cffi (auto.ru), Playwright (Avito).
- **Альтернативы:** браузер везде (медленно); HTTP везде (Avito банит).
- **Последствия:** два механизма поддержки, каждый оптимален для своей площадки.

#### ADR-3: Human-like поведение как первоклассное требование
- **Статус:** Accepted
- **Контекст:** заказчик требует снизить риск блокировок ценой скорости.
- **Решение:** джиттер интервала, паузы между конфигами, медленные браузерные действия, ротация UA, лимит страниц.
- **Последствия:** свежесть уведомлений вторична к безопасности сбора.

#### ADR-4: SQLite вместо PostgreSQL
- **Статус:** Accepted
- **Контекст:** один процесс, один пользователь, нужна портируемость.
- **Решение:** SQLite, один файл.
- **Последствия:** нет конкурентного доступа (не нужен); zero-config; идентичен local/server.
